# s13 Reflection

我的感受是，S13 和上一章一样，都有一些简单的状态校验，用来让操作在重复进入时尽量保持可解释；但这一章新增的重点是并发状态保护。后台任务本质上是另外启动了线程，而主 loop 和后台 worker 会同时读写任务数量、任务状态和任务结果，所以需要用锁保护这些共享数据。

慢任务判断目前还是很粗的教学实现：要么依赖模型显式传入 `run_in_background`，要么通过 `install`、`build`、`pytest` 这类字符串规则判断。这种方式简单有效，但不算完全可靠，因为真实命令耗时和业务语义不一定能靠关键词准确判断。

我也注意到它没有真正的等待后台任务机制。当前实现是在工具轮次结束时顺手收集“已经完成”的后台任务；如果任务还没完成，就继续留在后台状态里，等后续轮次再被发现。也就是说，本章重点不是等待，而是不阻塞主 loop。

现实中的后台任务系统往往会复杂很多：不同业务场景可能需要不同的超时机制、取消策略、输出保留策略或失败舍弃策略；多个后台任务之间也可能不是随便并行，而是要通过队列、优先级或者 DAG 管理先后顺序；同时还需要考虑并行数量上限、资源隔离、任务恢复和完成后如何主动通知主循环。

所以我把 S13 理解成后台执行机制的最小骨架：先证明慢工具可以被派发到后台，并通过 `bg_id` 和 `<task_notification>` 回到 Agent Loop。它解决的是“不要干等慢工具”这个核心问题，但距离完整的后台任务调度系统还有明显距离。
