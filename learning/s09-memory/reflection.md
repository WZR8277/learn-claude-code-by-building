# s09 Reflection

我的感受是，这个记忆管理有一些简单但有效的机制。它和 skill loading 有相似之处：不是把所有记忆一次性塞进上下文，而是先让索引常驻，再根据当前问题渐进式加载相关记忆。

本章的实现也比较直观：用 Markdown 文件存储记忆，用 `MEMORY.md` 做索引；提取记忆时会读取用户最近的几轮对话，并要求模型输出结构化的 `{name, type, description, body}`。相关记忆的查询主路径依赖模型选择，失败时用简单关键词匹配；记忆过多时则让模型做整理和淘汰。

这套流程完全嵌入 Agent Loop：开始时查记忆、运行时带入相关记忆、结束后写记忆、数量过多时整理记忆。整体设计比较容易理解，也能体现长期记忆和上下文压缩之间的关系。

不过现实系统里还有很多可以优化的地方。比如直接按数量触发整理比较粗糙，记忆是否失效也不应该只由数量决定；整理记忆如果阻塞主循环，也会影响用户体验。真实系统可能还需要更细的权重、时间、来源可信度、冲突处理、后台整理和可观测机制。
