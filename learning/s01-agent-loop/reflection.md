# s01 Reflection

## 原始观点

Claude Code 的 Agent Loop 本质上还是借鉴了 ReAct 模式的风格，可以说它是一种更加优化的基于 Tool 的 ReAct 架构。优势是方便并行调用工具、避免复杂文本解析、有严格的工具结果回执从而更加稳定。

并且注意到 Agent Loop 是基于是否需要工具调用去终止的，其本质原因在于，我们期待一个好的模型如果不需要工具调用应该在一次响应里给用户想要的结果，否则说明它需要通过调用工具进行信息外援。

## 整理后的理解

s01 让我意识到 Claude Code 的 Agent Loop 并不是抛弃 ReAct，而是把 ReAct 从自由文本约定升级成结构化工具协议。传统 ReAct 依赖 `Action:`、`Observation:` 这样的文本格式，外部程序需要解析模型输出才能知道要调用什么工具；Claude 的 tool loop 则直接返回 `tool_use` block，工具名、参数和调用 ID 都是机器可读字段。

这种结构化协议的优势主要有三点。第一，它避免了复杂和脆弱的文本解析。第二，它天然支持一次模型响应里出现多个工具调用，每个调用都可以独立执行。第三，它通过 `tool_use.id` 和 `tool_result.tool_use_id` 建立严格回执关系，让模型能稳定知道哪个工具结果对应哪个请求。

我也理解了 s01 的终止条件：Agent Loop 不是为了让模型在外部无限“多想几轮”，而是为了在模型需要外部信息或外部动作时提供工具结果。如果模型没有请求工具，说明 harness 没有新的外部观察要补充，控制权就应该回到用户。一个能力足够的模型应该在不需要工具时，在同一次响应里完成回答；如果它需要更多事实，才应该通过 `tool_use` 请求外援。

## 本地实现选择

我没有把全部代码写进一个文件，而是拆成：

- `cli.py`: 负责交互、环境变量加载和输出最终文本。
- `loop.py`: 负责 Agent Loop 协议和模型调用。
- `tool.py`: 负责工具定义和本地 bash 执行。

拆分之后，s01 的核心协议更容易单独测试。后续章节扩展工具注册、权限、hooks 时，也可以沿着这些边界继续演进。

## 上游简化点

s01 仍然是教学性质的最小版本。当前 bash 工具只做了非常粗粒度的危险命令拦截，还没有真正的工作区沙箱、权限审批、工具注册表、结构化错误分类或上下文压缩。这些边界会在后续章节逐步补齐。
