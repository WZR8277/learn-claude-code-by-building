# s15 Reflection

这一章比上一章清晰一些。它至少没有把子任务执行过程和主任务上下文直接搅在一起，而是通过文件邮箱模拟子任务与主任务之间的消息通信。

我的理解是：Lead 需要模型自己决定调用 `spawn_teammate` 工具来发起队友任务；队友跑在后台线程里，有自己的上下文和简化工具集。队友完成后把结果写进 Lead 的邮箱，主程序再通过大约一秒一次的轮询发现新消息，把消息注入主 Agent 的上下文，让模型基于新的执行进度判断下一步。

这个机制已经有了简单的通信协调：每个子任务都有自己的邮箱，也可以收到其他任务或 Lead 发来的消息；主程序通过锁避免异步唤醒和用户输入同时修改同一份 history。相比直接共享上下文，这种边界清晰很多。

但它仍然比较粗糙。用 `sleep(1)` 轮询查看是否有队友或后台任务完成，看起来不够优雅；如果任务之间有依赖顺序，或者 Lead 明确需要等待某个队友完成，目前也没有规范的等待机制。基于 JSONL 文件的消息管理很直观，但协议、并发写入、消息状态、失败重试和顺序保证都还不完善。

总体感受：S15 展示的是多 Agent 协作的最小形态，也就是“后台队友线程 + 文件消息总线 + Lead 唤醒注入”。它让模型可以利用异步队友结果继续决策，但距离真实工程里的团队协议、依赖管理、权限冒泡、取消超时和可靠消息系统还有明显距离。
