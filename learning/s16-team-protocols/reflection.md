# s16 Reflection

这一章我的理解是：对于协调子任务停止，实现了一个简单的两次握手；对于计划审批，则是正常的请求和回执批准。这两个都是常见的需要协议、需要回执匹配的场景。

协议里，请求 ID、发送方、接收方、处理状态、时间戳、负载等字段是有必要的。消息本身也至少应该包含请求 ID、发送接收方、消息类型、是否批准等数据。这样才能把一次普通通信升级成可追踪的任务管理机制，并避免一些危险操作只靠自然语言文本判断。

不过我感觉市面上应该已经有很多成熟方案了。这里的难点依旧更像传统后端问题：数据结构怎么设计，通信协议怎么约定，幂等怎么处理，优雅关停怎么做，状态机怎么避免乱跳。只是在 Agent 场景里，接口的一端变成了模型，协议消息也会被模型读到和生成。

整体来说，S16 让我看到 Agent Team 不只是“多个模型聊天”，而是要把请求、响应、状态和生命周期都协议化。否则多 Agent 一旦涉及关停、审批、权限和任务协作，就很容易退化成不可追踪的文本消息。
